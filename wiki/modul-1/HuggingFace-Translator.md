## Introduction

Hugging Face is a technology company and community hub that provides the infrastructure for collaborative machine learning. It functions primarily as a central repository where developers can host, share, and version-control open-source models, datasets, and demo applications. By providing standardized libraries—most notably the Transformers library—the platform enables users to integrate sophisticated neural networks into their workflows with minimal friction, regardless of whether they are working on natural language processing, computer vision, or audio tasks.

Beyond its role as a storage provider, the platform establishes a framework for reproducibility and transparency in AI development. Through the use of integrated tools for fine-tuning and evaluation, it allows researchers to build upon existing work rather than starting from scratch. Consequently, it has become the industry standard for managing the lifecycle of machine learning projects, facilitating the transition from experimental code to functional, deployed software.

You can access and explore HuggingFace from: [https://huggingface.co/](https://huggingface.co/)

MTUOC-server can start several models hosted at HuggingFace to perform translations (or other related tasks). In the examples provided in this tutorial we will explore only the task of translation. In other tutorials we will learn how to perform automatic post-edition and how can be explore models for other tasks.

Before exploring the HuggingFace translator in MTUOC-server you may want to explore several simple scripts for translating with several different models available at HuggingFace: [Hugging Face translation scripts](https://github.com/mtuoc/MTUOC-Machine-Translation-Practical-Course/wiki/Hugging-Face-translation-scripts)

## General configuration

You should first state in config-server.yaml that you want to use the HuggingFace models for translation:

```
MTengine: HuggingFace
```

Then you should edit the config-HFTranslator.yaml:

```
HFTranslator:
  model: "BSC-LT/salamandraTA-7b-instruct"
  template: "Translate the following text from English into Spanish.\nEnglish: {SLsegment} \nSpanish:" 
  #should include {SLsegment}
  extract_regex: None
  json: False
  device_map: "auto"
  trust_remote_code: true

generation_params:
  max_new_tokens: 400
  do_sample: true
  temperature: 0.7
  top_p: 0.95
  repetition_penalty: 1.2
  no_repeat_ngram_size: 3
  num_beams: 1
```

The first important thing to decide is which **model** you want to test, as well as the size of the model if some sizes are available. We can here distinguish two kind of models: General models and models specifically fine-tuned for translation. General models are able to perform a lot of different tasks, translation among them. Fine-tuned for translation are able only to translate and perform some other related tasks, as they have forgotten to perform general tasks. You can try any model but we provide a list of useful models available in HuggingFace:

**General models:**

* Llama3
* Mistral

**Models fine-tuned for translation**

* SalamandraTA
* TowerInstruct

The second most important parameter is the **template**, that is the prompt or instruction we will give to the model to perform the translation. Keep in mind that before sendint the prompt to the model the MTUOC-server will replace **{SLsegment}** with the actual source language segment to translate. Regarding the prompt, keep in mind that the results may vary depending on the prompt and that the fine-tuned for translation models may require an specific prompt.

The **extract_regex** parameter is used to get the actual translation for the answer. It's not always required. The **json** parameter is used if you ask the model to get the answer in json to indicate which parameter holds the translation.

The parameter **device_map** leverages the accelerate library to intelligently orchestrate model deployment across available hardware. By setting it to "auto", the system automatically calculates the most efficient distribution of model layers, prioritizing GPU VRAM and offloading to CPU RAM or even disk if the model size exceeds the available memory. This is essential for running large models like Salamandra-7B on consumer-grade or multi-GPU setups.

**trust_remote_code** Certain models on Hugging Face feature custom architectures or unique logic not yet integrated into the official transformers library. This flag grants the library permission to execute the model's own Python code locally. It is a mandatory requirement for specialized models (such as those from the SalamandraTA or TowerInstruct families) to ensure the neural network is instantiated exactly as the authors designed it.

**max_new_tokens** This establishes a definitive ceiling for the output length, specifically counting only the tokens generated by the model (excluding the input prompt). A value of 400 is generally sufficient for paragraph-level translations while preventing the model from entering infinite loops.

**do_sample** By enabling sampling, the model departs from a deterministic "greedy" approach (always picking the single most likely word) and instead chooses from a probability distribution. This introduces linguistic variety and leads to more natural-sounding, less repetitive translations.

**temperature** This parameter modulates the "creativity" of the output by scaling the logits. A value of 0.7 is considered a balanced middle ground: it flattens the distribution enough to avoid robotic phrasing but keeps it focused enough to ensure the translation remains accurate and contextually relevant.

**Low Temperature (0.1 – 0.4): Deterministic and Precise** When the temperature is set to a lower value, the model becomes more risk-averse. It significantly amplifies the probability of the most likely token while suppressing all other alternatives.
* Behavior: The output is highly predictable, consistent, and focused. It follows the most probable path, which is ideal for tasks requiring high factual accuracy or structural integrity.
* Best for: Technical translation, code generation, mathematical reasoning, and formal documentation.
* Risk: Setting it too low (near 0) can lead to "repetitive loops" where the model gets stuck in a cycle of the same phrases because it refuses to explore less probable but more natural variations.

**Moderate Temperature (0.5 – 0.8): Balanced and Fluid** This is often considered the "Goldilocks zone" for general-purpose AI interactions. It allows for enough variety to sound human without losing the original intent of the prompt.
* Behavior: The model occasionally selects the second or third most likely token, leading to more diverse vocabulary and varied sentence structures.
* Best for: General translation, creative writing, and conversational agents.

**High Temperature (0.9 – 1.5+): Creative and Erratic** As the temperature increases, the probability distribution becomes "flatter." The gap between the most likely word and the least likely word narrows, making the model much more exploratory.
* Behavior: The model becomes highly creative, imaginative, and unpredictable. It is more likely to use rare words or unconventional phrasing.
* Best for: Poetry, brainstorming, fiction writing, or "hallucinating" new ideas where accuracy is secondary to novelty.
* Risk: At very high values, the model may become incoherent or "hallucinate" nonsensical facts, as it starts choosing tokens that have almost no logical connection to the preceding context.

**top_p** (Nucleus Sampling) This filter restricts the model’s choices to the smallest set of tokens whose cumulative probability reaches 95%. It effectively "prunes" the long tail of low-probability, nonsensical words, ensuring that even when sampling is randomized, the output remains coherent.

**repetition_penalty** This applies a discount to the scores of tokens that have already appeared in the text. A penalty of 1.2 discourages the model from getting stuck in repetitive loops or reusing the same adjectives excessively, which is a common quirk in Causal LM architectures.

**no_repeat_ngram_size** This is a hard constraint that prevents the model from generating any sequence of three consecutive words more than once. It is a highly effective "fail-safe" for maintaining structural diversity in longer translations.

**num_beams** Setting this to 1 signifies that the model is performing a single-path search rather than "Beam Search." In the context of the other sampling parameters above, it means the model is prioritizing a faster, more fluid stochastic generation rather than an computationally expensive search for the most probable overall sequence.
